# Architecture Review — 2026-06-06

A structural review of FileOctopus: what is fine and should be preserved, what
should be fixed, and what can be improved later. Findings are grounded in code
metrics and a whole-repo dependency graph (871 files → 6,435 nodes, 14,943
edges, 346 communities) generated for this review, cross-checked against the
source.

This review supersedes parts of [`review-2026-05.md`](./review-2026-05.md) and
follows up on the still-incomplete
[Decompose Monolith Modules Plan](../archive/2026-05-17-decompose-monolith-modules.md).

## TL;DR

The core architecture is sound — the IPC trust boundary, the acyclic crate
layering, the planned-jobs model, and the stable error-code vocabulary are all
correct and consistently applied. The problems are not in the design; they are
in **enforcement and follow-through**:

1. The IPC contract is mirrored across four hand-maintained places with no
   codegen. (highest risk)
2. Privileged, subprocess-spawning logic has leaked into the "thin" Tauri shell.
3. Several monolith files from the decompose plan were never split.
4. The frontend shell has a prop-explosion / god-component coupling problem.
5. Two migrations (localStorage→SQLite, command-registry) are stuck half-done.

---

## What is fine — preserve it

| Area                                   | Why it's healthy                                                                    | Evidence                                                                                          |
| -------------------------------------- | ----------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------- |
| **IPC trust boundary** (ADR-0002/0003) | One validated type crosses the boundary; UI is a consumer, not a co-owner.          | `FileEntryDto` is the #1 god node (150 edges across 36 communities); 114 TS importers vs 24 Rust. |
| **Crate layering**                     | Acyclic, dependencies point downward onto pure-domain `vfs`.                        | Provider/config/terminal communities (C2, C5, C7, C13) have near-zero cross-community edges.      |
| **Planned-jobs model**                 | Cancellation, progress events, and history rows are consistent across mutating ops. | `jobs` + `app-core::runtime` + `OperationHistoryRepository`.                                      |
| **Stable error codes**                 | UI can branch on `FileOperationError::code()` / `VfsError::code()`.                 | `operationErrorMessage` in `OperationDialogView.tsx`.                                             |
| **Provider abstraction**               | `remote-core` trait + per-protocol crates is the right extensibility shape.         | sftp/smb/s3/gdrive/dropbox/onedrive crates.                                                       |
| **Schema migrations**                  | Forward-migration machinery, tested — not legacy cruft.                             | config at **v14**, tested upgrades v4/v6/v7/v13→current.                                          |

Do not "fix" the migration machinery (item above) or the god-node status of
`FileEntryDto` — both are working as intended.

## What to fix

### FIX-1 — Generate the IPC types instead of hand-mirroring them _(highest leverage)_

**Problem.** The IPC contract is mirrored by hand across four locations:

- **92** Tauri command handlers (`#[tauri::command]`)
- **177** DTO structs in `crates/app-ipc`
- **93** entries in `packages/ts-api/src/commandMap.ts`
- **`packages/ts-api/src/types.ts`** — **1,397 lines, fully hand-written** (no
  `ts-rs` / `specta` / `tsify` anywhere in the workspace)

The CLAUDE.md invariant ("update the DTO, `types.ts`, the client, the
commandMap, and the handler together") is enforced **only by discipline**. Given
`FileEntryDto`'s blast radius (114 consumers across ~30 frontend communities), a
single missed field rename drifts silently: TS still compiles, serde still
deserializes, and a `can_*` capability flag reads `undefined` — mis-gating a UI
action with no error.

**Direction.** Introduce Rust→TS type generation (`ts-rs` or `specta`) so
`types.ts` is generated from the `app-ipc` DTOs in CI. Ideally generate the
command name map too. This converts a discipline-enforced invariant into a
compiler-enforced one and removes ~1,400 lines of hand-maintained mirror.

**Tasks**

- [x] FIX-1.1 — Spike `ts-rs` vs `specta` on `app-ipc::fs` DTOs; pick one (ADR). → **ts-rs**, see [ADR 0005](../adr/0005-ipc-type-generation.md).
- [ ] FIX-1.2 — Derive type exports on all 177 DTOs; emit to a generated module.
- [ ] FIX-1.3 — Replace hand-written `types.ts` with the generated output; keep the `@fileoctopus/ts-api` public surface stable.
- [ ] FIX-1.4 — Generate or validate `commandMap.ts` against registered handlers.
- [ ] FIX-1.5 — Add a CI check that fails if generated types are stale (`git diff --exit-code`).

### FIX-2 — Move privileged logic out of the thin shell

**Problem.** `apps/desktop-tauri/src-tauri/src/commands/network.rs` is **1,298
lines** and contains real platform logic that should live in a crate:

- LAN discovery that **spawns subprocesses** — `command_output_with_timeout`,
  `parse_dns_sd_browse`
- cloud-root detection — `detect_cloud_storage_roots_from`, `cloud_entries`
- neighborhood / virtual-entry construction — `virtual_entry`,
  `root_neighborhood_entries`, `lan_neighborhood_entries`

This contradicts the project principle that the shell is thin and crates own
privileged platform operations. It is also a **security concern**: subprocess
spawning sits _outside_ the crate boundary where the rest of the platform ops —
and their tests — live.

**Direction.** Extract discovery/neighborhood logic into a crate (new
`discovery` crate, or fold into `remote-core`). The handler should validate
input and marshal DTOs, not run `dns-sd`.

**Tasks**

- [ ] FIX-2.1 — Carve discovery (`discover_lan_entries`, `parse_dns_sd_browse`, `command_output_with_timeout`, cloud-root detection) into a crate with its own tests.
- [ ] FIX-2.2 — Reduce `network.rs` to handler glue (validate → call crate → DTO).
- [ ] FIX-2.3 — Add unit tests for the subprocess timeout/parse paths at the crate level.

### FIX-3 — Split the remaining monolith files

The decompose plan stalled. Current offenders (non-test lines):

| File                                                       | Lines | Split hint                                                                                                   |
| ---------------------------------------------------------- | ----: | ------------------------------------------------------------------------------------------------------------ |
| `crates/config/src/lib.rs`                                 | 1,838 | `UserPreferences` impl ~700 lines (L656–L1364); move `PreferencesRepository` + the big impl out of `lib.rs`. |
| `crates/vfs/src/lib.rs`                                    | 1,462 | split domain types by concern (uri / entry / capabilities / operations).                                     |
| `commands/network.rs`                                      | 1,298 | covered by FIX-2.                                                                                            |
| `crates/app-ipc/src/lib.rs`                                | 1,245 | per-domain modules exist; `lib.rs` still carries conversions + `error_codes`.                                |
| `packages/frontend/src/app/FileOctopusApp.tsx`             | 1,147 | see FIX-4.                                                                                                   |
| `packages/frontend/src/app/providers/TerminalProvider.tsx` | 1,019 | extract session/profile logic from the provider.                                                             |
| `packages/frontend/src/hooks/useAppInit.ts`                |   828 | startup + all migration shims in one hook.                                                                   |

**Tasks**

- [ ] FIX-3.1 — `config/lib.rs`: extract `PreferencesRepository` and `impl UserPreferences` into modules.
- [ ] FIX-3.2 — `app-ipc/lib.rs`: relocate conversions/error_codes into dedicated modules.
- [ ] FIX-3.3 — `vfs/lib.rs`: split by concern.

### FIX-4 — Decouple the frontend shell (prop explosion)

**Problem.** The graph's weakest-cohesion large communities are all frontend
shell code, with more external than internal edges:

- **C3 File Panel Props & Dialogs** — 94 nodes, **446** cross-community edges
- **C0 Status Bar & Mutation State** — 156 nodes, 320 cross edges
- **C6 FileOctopusApp Shell Root** — 171 cross edges

`FileOctopusApp.tsx` + `useAppInit.ts` + `panelStore.ts` form a tightly
cross-wired hub that resists isolated change — the "shell-context prop-explosion"
flagged in the 2026-05 review, still present.

**Tasks**

- [ ] FIX-4.1 — Inventory the props drilled through `FileOctopusApp`; group into cohesive contexts/stores.
- [ ] FIX-4.2 — Move startup migrations out of `useAppInit` into a dedicated migration module.
- [ ] FIX-4.3 — Split `FileOctopusApp.tsx` into layout + wiring; push panel state into stores.

## What to improve (lower urgency)

### IMP-1 — Finish the two in-flight migrations

- **localStorage → SQLite config.** ~16 frontend files still read `localStorage`
  directly, bridged by live shims (`migrateLegacyChromePreferences`,
  `viewModeMigration`, `migrateLegacyToolbarEntries`). Set an end date, verify no
  installed client holds old-format state, then delete the shims.
- **Command registry.** `PALETTE_LEGACY` in `paletteEntries.ts` holds commands
  not yet migrated to the registry; `LEGACY_COMMAND_ALIASES` aliases renamed IDs.
  Finish the migration or document the split as intentional.

### IMP-2 — Orphan / dead-code pass

The graph reports **1,357 weakly-connected nodes**. Many are doc/concept nodes,
but the set is worth a sweep for genuinely orphaned code.

### IMP-3 — Consolidate duplicate state files

`CRON_STATUS.md` / `CRON_TASKS.md` exist at both repo root and `docs/plans/`.
Pick one home.

---

## Suggested order

1. **FIX-1** (IPC type generation) — biggest risk reduction, also shrinks the codebase.
2. **FIX-2** (extract `network.rs` logic) — restores the boundary, closes the subprocess-outside-crate gap.
3. **FIX-4** (frontend shell decoupling) — unblocks isolated frontend change.
4. **FIX-3 / IMP-\*** — steady cleanup.

## Method note

Metrics are from `wc -l` / `grep` over the working tree at commit time and a
whole-repo graphify run (`graphify-out/`). Graph-derived claims (god nodes,
cross-community edge counts, weakly-connected count) reflect that snapshot;
verify against current source before acting on any individual item.
