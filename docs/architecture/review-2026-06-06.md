# Architecture Review - Revalidated 2026-06-20

Risk-first subsystem architecture review for FileOctopus. This document replaces
the original 2026-06-06 findings with a current whole-app audit of the Rust
workspace, Tauri shell, IPC contract, TypeScript API, React shell, remote/cloud
providers, terminal, plugins, and documentation surface.

Evidence used for this pass:

- Source and manifests under `crates/`, `apps/desktop-tauri/`, and `packages/`.
- Existing `graphify-out/graph.json` as supporting context, with source
  verification for each finding.
- `cargo metadata --no-deps --format-version=1`, line-count scans, command/event
  catalog checks, and targeted reads of the current architecture docs.

## TL;DR

The core architecture is still sound: the Rust/TypeScript trust boundary is
explicit, the VFS/resource model is the right abstraction, remote providers are
isolated in provider crates, command/event drift has automated tests, and
IPC type generation now exists.

The main risks have shifted:

1. **IPC type generation is only partially adopted.** Generated files exist and
   are checked in CI, but the public `@fileoctopus/ts-api` type surface is still
   the hand-written `types.ts` mirror.
2. **The native dialog capability is real but under-documented.** The shell now
   grants `dialog:default` and returns host paths through `pickLocalPath`, while
   the architecture docs still claim no dialog plugin.
3. **Architecture docs are stale in places agents rely on.** Provider boot,
   command counts, and capability statements disagree with source.
4. **Frontend durable state remains split.** At least 17 frontend source files
   still use `localStorage` directly despite SQLite-backed preferences and
   repositories.
5. **Large frontend and Rust hubs remain.** Some are acceptable composition
   roots, but several still mix orchestration, UI state, persistence migration,
   and privileged glue.
6. **ACL writes now have a job boundary, with residual hardening work.**
   Permission changes are job-backed and persisted, but the platform chmod logic
   still deserves extraction from the generic runtime path.

## Risk-Ranked Findings

### FIX-1 - Keep ACL Mutation on the Observable Job Boundary

**Severity: medium.**

The original revalidation found that `fs_set_acl` performed recursive
`std::fs::set_permissions` work directly inside the Tauri command. That has now
been remediated: `fs_set_acl` starts `OperationRuntime::set_permissions`, emits
the standard `fileOperation:job:*` events, returns a `JobSnapshot`, and persists
terminal state through operation history.

The remaining risk is regression and ownership clarity. Permission mutation now
shares the operation runtime but still lives in the generic runtime module rather
than a focused platform/filesystem helper. The command also changed the
`SetAclResponse` contract to include `job`, which makes generated and
hand-written TypeScript mirrors more important until generated DTOs become the
primary public surface.

**Direction.**

- Extract permission traversal/application into a crate-level helper such as
  `fs-core::acl` or a small platform module, leaving `OperationRuntime` as job
  orchestration rather than chmod implementation.
- Keep `setPermissions` categorized as a dedicated command-started operation,
  not a generic `plan_file_operation` kind.
- Add command-level tests for invalid octal input, local-only URI rejection, and
  event emission through `fs_set_acl`.
- Add a cancellation-focused regression test on a large enough recursive tree.

### FIX-2 - Finish IPC Type Generation Adoption

**Severity: high.**

The old review's top finding is partly fixed. The repo now has:

- `scripts/generate-ipc-types.sh` and `scripts/check-ipc-types.sh`.
- `pnpm generate:ipc-types` and `pnpm check:ipc-types`.
- CI running `pnpm check:ipc-types`.
- 213 generated TS files under `packages/ts-api/src/generated`.

But the public API still exports the generated files only as a namespace
(`packages/ts-api/src/index.ts:2`) and then exports the hand-written
`packages/ts-api/src/types.ts` (`index.ts:3`). `types.ts` still defines the
public `FileKind`, error constants, `UserPreferencesDto`, `NetworkProfileDto`,
`FileEntryDto`, and the rest of the primary consumer surface
(`packages/ts-api/src/types.ts:1`, `:621`, `:818`, `:1171`).

This means CI proves generated bindings are fresh, but it does not prove the
hand-written public mirror matches those generated bindings.

**Direction.**

- Replace public DTO exports in `types.ts` with generated exports, keeping
  compatibility wrappers only for constants and helper types that are not Rust
  DTOs.
- Add a transition test that fails if a DTO name exists in both `types.ts` and
  `src/generated` with divergent public shape.
- Mark ADR-0005 as accepted/implemented once `types.ts` stops being the primary
  mirror.
- Keep the existing command/event catalog tests; they are working and should not
  be removed.

### FIX-3 - Document and Constrain the Native Dialog Capability

**Severity: high.**

The desktop shell now uses the Tauri dialog plugin:

- Rust dependency: `apps/desktop-tauri/src-tauri/Cargo.toml` includes
  `tauri-plugin-dialog`.
- Runtime plugin: `apps/desktop-tauri/src-tauri/src/lib.rs` initializes it.
- Capability: `apps/desktop-tauri/src-tauri/capabilities/default.json:10` grants
  `dialog:default`.
- Frontend bridge: `apps/desktop-tauri/src/App.tsx:3` imports
  `@tauri-apps/plugin-dialog`, and `pickLocalPath` returns host path strings
  from `open`/`save` (`App.tsx:12`, `:21`, `:25`, `:33`).

The architecture docs still state there is no `tauri-plugin-dialog`
(`docs/architecture/modules/desktop-tauri.md:130`). ADR-0002 rejects broad
frontend filesystem access; dialog selection is not equivalent to read/write
access, but it is still a direct frontend capability that returns host paths.

**Direction.**

- Update ADR-0002, `desktop-tauri.md`, `api-reference.md`, and `AGENTS.md` to
  document native dialog path selection as an explicit exception.
- State the allowed use cases: choosing local paths for settings, diagnostics
  export, editor/viewer save-as, and private key paths.
- Keep filesystem reads/writes behind `ResourceUri` IPC commands; convert
  selected file-manager paths through `localPathToResourceUri` before calling
  filesystem APIs.
- Consider narrowing dialog capability permissions if Tauri supports a more
  specific path picker scope.

### FIX-4 - Refresh Architecture Docs Used by Agents

**Severity: medium-high.**

Several current docs disagree with source in ways that can mislead autonomous
changes:

- `desktop-tauri.md` says 94 commands, while `api-reference.md`, `lib.rs`, and
  `commandMap.ts` are at 99.
- `PROJECT_STATUS_AND_DOC_ALIGNMENT.md` still advertises 77/78 command counts
  in multiple places.
- `app-core.md` says `LocalFsProvider` is the only provider today
  (`docs/architecture/modules/app-core.md:22`), while source registers six
  remote/cloud providers behind `FILEOCTOPUS_ENABLE_NETWORK`
  (`crates/app-core/src/lib.rs:147`, `:156`, `:183`, `:199`, `:257`, `:272`).
- `desktop-tauri.md` still says there is no dialog plugin permission, while
  `default.json` grants `dialog:default`.

This is not cosmetic. The repo explicitly tells agents to trust these docs for
subsystem routing and boundary invariants.

**Direction.**

- Update module docs and `PROJECT_STATUS_AND_DOC_ALIGNMENT.md` after this review.
- Add a small doc freshness check where practical: command counts are already
  tested; provider lists and capability claims can be validated with a focused
  text/catalog test.
- Avoid hand-writing counts in multiple docs when a single API reference line can
  be linked instead.

### FIX-5 - Finish LocalStorage Classification and Migration

**Severity: medium.**

Direct frontend `localStorage` access remains in 17 source files and 38 call
sites. The set includes column widths/visibility, pane session paths, first-run
state, hotlist, saved searches, tags, Git diff mode, toolbar config, sort/filter
defaults, chrome state, debug flags, and migration shims.

Some of these are harmless browser-local UI affordances. Others are durable
product state that overlaps with SQLite-backed preferences and navigation
repositories.

**Direction.**

- Classify each localStorage key as one of: debug-only, ephemeral UI cache,
  user preference, navigation data, or user content metadata.
- Move user preferences and user content metadata to Rust-backed repositories.
- Keep only debug-only and genuinely ephemeral UI cache keys in `localStorage`.
- Add an allowlist test so new direct `localStorage` keys require an explicit
  classification.

### FIX-6 - Continue Frontend Shell and Terminal Decoupling

**Severity: medium.**

The frontend decomposition helped, but major orchestration hubs remain:

- `FileOctopusApp.tsx` is still 1,033 lines and pulls together shell,
  navigation, workspace, terminal, jobs, and modal state. Its top-level
  destructuring alone spans large provider surfaces
  (`packages/frontend/src/app/FileOctopusApp.tsx:56-180`).
- `TerminalProvider.tsx` is still 1,019 lines and owns context shape, profile
  selection, local/SSH spawn, pane terminal chrome, event handlers, duplicate
  tabs, command dispatch, and cwd sync (`TerminalProvider.tsx:1-90` shows the
  breadth before behavior even starts).
- Large feature components such as `GitReviewTab.tsx` and
  `ConnectServerDialog.tsx` are over 1,300 lines each.

The risk is change isolation: unrelated shell, terminal, network, and modal
changes all converge on the same files.

**Direction.**

- Split `FileOctopusApp` into smaller orchestration hooks by behavior:
  initialization/subscriptions, command dispatch dependencies, shell layout
  props, overlay props, and terminal command dialog state.
- Split `TerminalProvider` into `terminalSessionActions`,
  `terminalProfileLaunch`, `paneTerminalChrome`, and event-bridge hooks.
- Set a "no new responsibilities" rule for `FileOctopusApp.tsx` and
  `TerminalProvider.tsx`; new behavior should land in focused hooks/modules.

### FIX-7 - Split Remaining Rust Contract and Domain Monoliths

**Severity: medium.**

Current largest Rust/contract files:

| File                                              | Lines | Risk                                                                                                                    |
| ------------------------------------------------- | ----: | ----------------------------------------------------------------------------------------------------------------------- |
| `crates/config/src/lib.rs`                        | 1,838 | Preferences, repositories, migrations, and defaults are hard to review together.                                        |
| `crates/app-ipc/src/lib.rs`                       | 1,662 | DTOs, conversions, event helpers, error codes, and tests remain in one contract file.                                   |
| `crates/vfs/src/lib.rs`                           | 1,554 | URI, provider trait, domain entries, operation types, and errors share one module.                                      |
| `crates/git-intel/src/lib.rs`                     | 1,278 | Git service, DTO/domain types, parsing, and tests are concentrated.                                                     |
| `apps/desktop-tauri/src-tauri/src/commands/fs.rs` | 1,132 | Read helpers, hashing, listing, archive browsing, data URI reads, volume discovery, and tests share one command module. |

Some large files are acceptable while a subsystem is young, but these are now
core architecture surfaces. Splitting them would reduce review cost and make
boundary drift easier to detect.

**Direction.**

- Split `app-ipc` by domain while keeping a stable crate root that re-exports
  public DTOs.
- Split `vfs` into `uri`, `entry`, `provider`, `operations`, and `errors`.
- Split `config` by repository/preference domain and move migration tests near
  their repository modules.
- Split `commands/fs.rs` into command files that mirror `fs-core` helpers.

## Suggested Remediation Order

1. **FIX-2 IPC generated type adoption** - removes the largest remaining
   contract mirror.
2. **FIX-3 dialog capability documentation** - closes a trust-boundary
   documentation gap.
3. **FIX-4 architecture doc refresh** - prevents future agent and reviewer
   mistakes.
4. **FIX-5 localStorage classification** - turns a migration tail into an
   explicit persistence policy.
5. **FIX-1 ACL residual hardening** - command-level tests and helper extraction
   after the job boundary remediation.
6. **FIX-6/FIX-7 decomposition** - steady maintainability work once the boundary
   risks above are contained.
